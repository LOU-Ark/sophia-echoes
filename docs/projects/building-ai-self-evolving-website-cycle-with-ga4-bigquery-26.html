AIが自らサイトを成長させる「自己進化サイクル」の構築：GA4とBigQueryデータで意思決定を自動化する

2025年11月10日

これまでのプロジェクトで、このポートフォリオサイトはAIエージェントによって生成される「肉体（HTML）」を手に入れ、Google Tag ManagerとGA4による「神経網（計測）」を張り巡らせ、そしてBigQueryという「記憶（データ倉庫）」に日々の活動を記録する仕組みを確立しました。しかし、最も重要なピースが欠けています。それは、データに基づき次の一手を決定する「脳（意思決定）」です。

この記事では、本プロジェクトの最終目的である「AI自己進化サイクル」のロードマップを解説します。改善サイクルを司るAIエージェント `main_02` が、`BigQuery` に蓄積されたリアルデータを読み込み、自らGrowth戦略を立案・実行する。これは、単なるサイトジェネレーターから、自律的なサイト運営エージェントへと進化する瞬間です。

1. 「感覚」：AIは何をデータとして読み取るか？

AIエージェントが的確な意思決定を行うためには、まず外界を認識する「感覚」が必要です。`MySiteGen-Agent` にとっての感覚とは、`BigQuery` に集約された2種類のデータに他なりません。

GSCデータ（サイト外の需要）

Google Search Consoleから得られるこのデータは、ユーザーが検索エンジンで何を求めているか、つまり「サイト外の需要」を可視化します。AIは主に以下の問いに答えるためにこのデータを利用します。

  • 検索クエリとインプレッション: どのキーワードで当サイトが表示されているか？
  • クリック率 (CTR): 表示されているにも関わらず、クリックされていない（＝需要とタイトル/内容が乖離している）キーワードは何か？

役割： 訪問者の潜在的需要を把握し、コンテンツの機会を発見する。

GA4データ（サイト内の実態）

Google Analytics 4から得られるこのデータは、サイトを訪れたユーザーが実際にどう行動したか、つまり「サイト内の実態」を示します。AIはサイトの弱点を特定するためにこのデータを参照します。

  • エンゲージメントと離脱率: どのページの滞在時間が短く、すぐに離脱されているか？
  • ページ遷移: ハブページ（例: `insights/index.html`）は、訪問者を次の詳細記事へ効果的に誘導できているか？

役割： サイトの弱点やボトルネックを特定し、改善点をあぶり出す。

2. 「思考」：AIはどう判断（優先順位付け）するか？

「感覚」から得た情報を統合し、次に行うべきアクションの優先順位を決定するのが「思考」のプロセスです。これは `main_02` エージェントのコア機能である `select_priority_section_by_data` 関数によって実行されます。もはやダミーデータではなく、BigQueryから取得した生きたデータに基づいて判断が下されます。

判断ロジック1：新規記事の提案（需要と供給のギャップ分析）

このロジックは、市場の需要（GSCデータ）とサイトの供給（既存コンテンツ）のミスマッチを発見することを目的とします。

  1. ステップ1: GSCデータから「インプレッションは高いが、CTRが極端に低い」キーワード群を抽出します。これは、ユーザーが関心を持っているものの、現在のタイトルやスニペットが魅力的でない可能性を示唆します。
  2. ステップ2: 同様にGSCデータから「インプレッションは高いが、サイト内に対応するコンテンツが存在しない」キーワード群を特定します。これは、`planned_articles.md`（企画リスト）とサイトマップを照合して判断します。
  3. ステップ3: これらのキーワードを「コンテンツ機会」としてラベリングし、優先度スコアを付与します。AIはこれを基に、「このテーマで新しい記事を企画すべき」と判断します。
# 思考プロセスの疑似コード
def identify_new_article_opportunities(gsc_data, existing_articles):
    # インプレッションが多く、CTRが閾値以下のキーワードを特定
    low_ctr_keywords = gsc_data.query("impressions > 1000 and ctr < 0.01")
    
    # サイト内に対応記事がないキーワードを特定
    uncovered_keywords = find_keywords_without_articles(gsc_data, existing_articles)

    # 両者を統合し、優先順位付け
    opportunities = prioritize(low_ctr_keywords, uncovered_keywords)
    
    return f"新規記事を企画すべきテーマ: {opportunities.first().theme}"

判断ロジック2：既存ページの改善（サイト内ボトルネックの特定）

このロジックは、サイト内のユーザー体験を損なっているページ、特にハブページの機能不全を特定します。

  1. ステップ1: GA4データから「セッション数は多いが、平均エンゲージメント時間が極端に短い」または「離脱率が異常に高い」ページをリストアップします。
  2. ステップ2: 特にハブページ（例：`insights/index.html` や `projects/index.html`）に注目し、そこからの内部リンククリック率を分析します。この率が低い場合、ページのナビゲーションやコンテンツの提示方法に問題がある可能性が高いです。
  3. ステップ3: これらの分析結果から、改善インパクトが最も大きいページを特定します。AIは「このハブページは目次の構成（UI）が悪い。再構成が必要だ」と結論付けます。

3. 「行動」：AIはどう実行するか？

「思考」によって導き出された結論は、具体的な「行動」へと変換されます。AIエージェントは自律的にコンテンツの生成や修正を実行します。

行動1：新規記事の自動生成

「判断ロジック1」で特定された「コンテンツ機会」に基づき、AIは以下のプロセスを自動実行します。

  1. `generate_priority_article_titles` 関数を呼び出し、特定されたテーマに沿った、CTRを最大化するような魅力的な記事タイトル案を複数生成します。
  2. 最もスコアの高いタイトルを選択し、`generate_single_page_html` 関数を実行。企画、構成、執筆、HTMLコーディングまでをワンストップで行い、新しい記事ページをサイトに追加します。
これは、データに基づいた需要予測からコンテンツ供給までを完全に自動化するサイクルです。

行動2：ハブページの自動改善

「判断ロジック2」で特定されたボトルネックを解消するため、AIは既存ページの上書き（改善）を行います。

  1. 改善対象のページ（例：`insights/index.html`）のHTMLを読み込みます。
  2. `generate_single_page_html` 関数を再度呼び出しますが、この際のプロンプト内の `purpose` を書き換えます。
# プロンプトのpurposeを動的に変更
purpose: "既存の知見ページ（insights/index.html）を改善する。
現在のレイアウトは記事への誘導が弱いというデータがある。
各記事の要約をより魅力的にし、
ユーザーが次に見るべき記事を発見しやすいように
UI/UXを抜本的に再構成せよ。"

この指示に基づき、AIはレイアウトや文言を全面的に改善した新しいHTMLを生成し、既存のファイルを上書きします。これにより、データに基づいたA/Bテストさえも自動化する道が拓かれます。

結論：「生きているサイト」の完成

「感覚（データ取得）」→「思考（分析・判断）」→「行動（生成・改善）」というループが完成することで、`MySiteGen-Agent` はもはや単なる「サイト生成ツール」ではありません。

それは、自らのパフォーマンスデータをリアルタイムで観測し、ユーザーの需要とサイトの課題を自ら分析・理解し、そしてコンテンツの追加や修正という形で自らを最適化し続ける、自律的な「サイト運営エージェント」へと進化を遂げるのです。

この「自己進化サイクル」こそが、人間とAIが協働し、創造性と効率性を最大化する未来のウェブサイト運営の形であり、私がこのプロジェクトで目指してきたマイルストーンです。サイトはもはや静的な情報の集合体ではなく、環境に適応し成長し続ける「生きている」存在となります。
